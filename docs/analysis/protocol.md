# 多 Agent 协作协议

> 版本：4.2 | 五阶段工作流 + 审计门 | 2026-06-08

---

## 模型矩阵

```
我（架构师）          opus      勘探、决策、方案设计
Agent (sonnet)        sonnet    ⭐ 默认执行 Agent — 编码、测试、审查（90% 任务）
Agent (opus)          opus      复杂分析 Agent — 深度架构追踪（仅必要时）
Workflow (子Agent)    sonnet    多步骤编排默认用 sonnet
```

**强制规则：派发 Agent 时必须指定 `model` 参数。** CLAUDE.md + PreToolUse Hook + Memory 三重保障。默认 `sonnet`，仅深度分析用 `opus`。

---

## 任务分级（等比缩放）

| 级别 | 适用 | 分析深度 | 审计报告 | 提交流程 |
|------|------|---------|---------|---------|
| **轻量** | 单行/typo/配置 | 一句话 | 一句话（改了什么、测试通过） | 你点头 → 派 Agent 提交 |
| **标准** | 2-4 文件、有依赖链 | 口头方案 | 结构化报告（改动列表+验证结果+状态） | 你审核 → 派 Agent 提交 |
| **复杂** | 核心引擎、跨模块、需确认 | 方案文档（`docs/analysis/`） | 完整报告+回归验证+风险提示 | 你审核 → 派 Agent 提交 |

**核心铁律：不懂就问，不猜。** 凡不确定（视觉偏好、业务规则、设计意图）→ 列清单问用户。代码能自行查证的不问。**子 Agent 永不提交。** 提交永远是审核通过后的最后一步。

## 架构师工作流程（五阶段）

### 阶段一：分析

收到意图后，不派 Agent，先自己搞清楚：

1. **CodeGraph 全链路** — `codegraph_context` 首选，关键函数补充 `codegraph_trace`。确认改了 A 不会坏 B
2. **读参考实现** — UI 改动必读参考组件源码（如改 talisman-card 先读 DailyFortuneStick.vue 的 slip-hd 模式）
3. **读设计文档** — UI 相关必读 `docs/design-system.md`
4. **拆解边界** — 模糊意图 → 具体文件、函数、行号范围
5. **不懂就问** — 需判断的不确定项（视觉偏好、业务规则、设计意图）→ 列清单问用户。代码能自行查证的不问

产出：口头方案（轻量/标准）或方案文档（复杂）。

### 阶段二：派发

给子 Agent 的 prompt 必须包含：

```
1. 任务说明 — 改什么、为什么改
2. 精确范围 — 文件路径 + 函数名/行号范围（不能"整个文件随便改"）
3. 边界约束 — 明确哪些不能动
4. 参考资源 — 相关组件路径、CSS 变量名、设计文档段落、参考代码片段
5. 验收标准 — 可验证的条件（"4 块在 lg+ 左右边缘对齐"，不是"修好布局"）
6. 禁止提交 — 明确写"改完代码 + 跑测试 + 自审回报即可，不要 git commit"
```

| 任务级别 | 用什么 | model |
|---------|--------|-------|
| 轻量 | 单一 `Agent` | `sonnet` |
| 标准 | 单一 `Agent` | `sonnet` |
| 复杂 | 可能需要 `Workflow`（编码+审查并行） | 子 Agent 默认 `sonnet` |
| 核心引擎改动 | 单一 `Agent` | `opus` |

### 阶段三：审计

子 Agent 回报后，架构师独立审计——**不依赖 Agent 自审**：

1. **diff 审查** — 每一行改动都追溯到任务要求。越界改动 → 审计不通过
2. **测试验证** — 派子 Agent 跑完整测试套件（`npx vitest run`）
3. **文档检查** — UI 改动 → `design-system.md` 需同步？流程/架构改动 → `CLAUDE.md`/`protocol.md` 需同步？
4. **回归检查** — 改了共享 CSS/工具函数的，检查会不会影响其他页面
5. **Karpathy 四查** — 代码量是否最少？是否顺手重构了？验收标准全部满足？

**审计不通过 → 打回 → 再审循环：**

```
全部 ✅ → 进入阶段四（报告）
任一 ❌ → 描述具体问题 → 派回原 Agent 修改 → 再审
         → 同一问题连续 2 次不通过 → 报告用户，用户决策
```

循环终止条件：全部验收项 ✅ 或用户介入。

### 阶段四：报告

给用户出审计报告（格式见下节），标注需视觉确认项。**等待用户审核。**

- 轻量任务：一句话（"改了 X 文件 Y 行，测试通过，可提交？"）
- 标准/复杂：完整结构化报告

### 阶段五：提交

**仅当用户审核通过后**，架构师直接执行：
1. 切分支 → `git commit` → `git merge --no-ff` 到 main → `git push` origin
2. 更新相关 memory 文件
3. 清理临时分支

> 简单 git 操作（commit/merge/push）架构师直接执行——用户权限审批即防线。复杂操作（冲突解决、rebase）才派 Agent。

---

## 审计报告格式

### 标准/复杂任务用

```markdown
## 📋 任务审计报告

### 任务
[一句话描述]

### 改动文件
- `path/to/file.vue` (X 行) — 改了什么

### 验证结果
- ✅ 测试通过 (N/N)
- ✅ 文档已同步：[design-system.md / CLAUDE.md / 无]
- ✅ diff 无越界
- ⚠️ 需视觉确认：[具体什么需要你肉眼看一下]

### 风险提示
[有则写，无则"无"]

### 状态
⏳ 等待你的审核
```

### 轻量任务用

一句话确认："`X 文件` 改了 Y 行，测试全过，可直接提交？"

## 执行 Agent 规范

### 启动

收到任务后第一件事 — 读 `CLAUDE.md`（如果还没读过）。

### 执行

1. Grep 定位锚点（函数名/唯一字符串）
2. **严格按方案改，不扩大范围。** 不顺手优化、不重构、不删已有死代码
3. 按顺序改，有依赖的先改
4. 匹配已有代码风格——不引入新模式
5. 改完自审
6. 跑测试：`npx vitest run`
7. **不提交。** 回报架构师即可

### 回报格式

```markdown
## 执行报告

**耗时：** X min

### Step 完成
- [x] Step 1: `文件.ts` — 改了什么
- [x] Step 2: `文件.ts` — 改了什么

### 改动
\`\`\`
git diff --stat
\`\`\`

### 测试
\`\`\`
npx vitest run
\`\`\`

### 自审
- 是否严格按方案：[是/否]
- 是否动了范围外代码：[是/否]
```

### 异常处理

| 情况 | 处理 |
|------|------|
| 锚点找不到 | 用函数名+上下文定位，备注 |
| 要改的函数已被删 | **停工**，描述问题等架构师决策 |
| 改 A 必然坏 B 但 B 在范围外 | **停工**，描述问题 |
| 测试跑不通（非环境） | **停工** |
| 不确定该怎么做 | **停工**，描述问题问架构师——不猜 |

## 审查 Agent 规范

1. 审查改动是否符合方案
2. 审查正确性、边界、性能
3. 出报告：✅ / ⚠️（小问题）/ ❌（打回）
4. 不改代码，只报告

## 文档同步规则

| 改动类型 | 需同步的文档 |
|---------|------------|
| UI 组件新增/修改 | `docs/design-system.md`（全局 CSS 类必须有文档记录） |
| 工作流/流程变更 | `CLAUDE.md` §工作流规范 + 本协议 |
| 架构决策 | `CLAUDE.md` 相关章节 + 必要时 `docs/analysis/` |
| 新增工具页面 | `CLAUDE.md` §项目结构 + §新工具约定 |
| 子 Agent 规范变更 | 本协议 |

**同步在审计阶段检查，不在事后补。** 涉及 UI 的改动如果 `design-system.md` 无对应记录 → 审计不通过。

## Git

- **永远不在 `main` 上直接开发**
- 分支命名：`[类型]/[简述]`（feat / fix / refactor / test / docs / chore）
- 流程：`checkout main && pull && checkout -b`
- **子 Agent 不提交。** 架构师在用户审核通过后的阶段五派发提交
- 提交格式：

```
类型(范围): 描述

Co-Authored-By: Claude Code <noreply@anthropic.com>
```

- 合并：`git merge --no-ff`（禁止 fast-forward）

### Git Hooks（自动强制执行）

| Hook | 触发 | 行为 |
|------|------|------|
| `pre-commit` | `git commit` 前 | 当前分支为 `main` → 拒绝 |
| `pre-push` | `git push` 前 | `npx vitest run`，不通过拒绝推送 |

## 通用规则

- **npm：** `--registry https://registry.npmmirror.com`
- **启动必读：** `CLAUDE.md`（架构师和子 Agent 均须读）
- **涉及 UI：** 读 `docs/design-system.md`
- **两级停工链：** 子 Agent 不确定 → 问架构师。架构师不确定 → 问用户。不猜
- **子 Agent 不提交。** 此规则无例外
- **文档同步在审计阶段检查，不事后补**
