# 玄·道 — 项目全记录

## 项目起源

2026 年 5 月 24 日，项目开始。

起因是跟朋友聊天，问对玄学感不感兴趣。我自己平时就爱看看每天的运势什么的——算是"赛博算命"吧。朋友说也挺有兴趣，我就想着做个项目，同时练练 AI 辅助开发的能力。

算是一个兴趣驱动的练手项目，但做着做着就认真了。

## 规模

从 5 月 24 日到 6 月 7 日，**14 天**：

- 480 个 commit（非 merge），115 个 merge commit
- 64015 行代码（Vue + TypeScript + CSS）
- 80 个 .vue 文件，113 个 .ts 文件
- 62 个组件，21 个 composable，20 个常量文件
- 36 个测试文件，2043 测试用例
- 22 个功能分支，全部通过 `merge --no-ff` 合并

## 什么项目

一个传统命理推演平台，线上地址 xuanji.me。提供八字排盘、六爻占卜、生肖运势、星座星盘、测字、姓名测试、择日、称骨等 11 个工具。

技术栈：Nuxt 3 + Tailwind CSS + SQLite。部署：阿里云轻量应用服务器（成都），PM2 + Nginx 反代，域名配了 Let's Encrypt 自动续期的 HTTPS。PWA 支持，含 iOS 适配。

## 项目历程

### Phase 1：骨架搭建

从零搭了三样东西：

- **前端骨架**：Nuxt 3 + Tailwind CSS，根组件、路由、布局系统、登录注册页
- **后端 API**：Nitro + SQLite，认证系统（PIN 哈希用 scrypt + 16 字节随机盐）、session 管理、profile CRUD
- **设计系统**：墨韵风格设计令牌（墨 7 阶/纸 6 阶/朱砂/金/玉）、CSS 工具类、CSP 安全头
- **共享组件**：InkDivider（墨韵分割线）、ToolPageLayout（三栏布局）、PageHero、FortuneBars、骨架屏

两次审计修复：Phase 1 review findings round 1 + round 2。

### Phase 2：第一批工具（生肖 + 星座）

先做了两个最简单的工具验证整个技术栈：

- 生肖计算引擎 + 星座计算引擎（含测试）
- 两个工具页完整实现（子组件、侧边导航、运势图表）
- 安全加固：PIN 哈希（scrypt）、session 过期、安全日志、速率限制

### Phase 3：八字引擎

核心命理引擎，技术含量最高的部分：

- **节气计算**：24 节气精确日期，用于月柱判定（不用硬编码 `day < 4`）
- **四柱推算**：年柱、月柱、日柱、时柱，含五虎遁月、五鼠遁时
- **十神系统**：天干十神矩阵，日主不能返回"日主"（只能返回"比肩"）
- **大运排法**：阳男阴女顺排、阴男阳女逆排，子平法计算起运年龄
- **纳音五行**：天干地支索引同奇偶校验
- **日期解析**：用 `parseDate()` 替代 `new Date()`，避免时区问题
- **农历转换**：lunar-javascript 库，农历转公历后计算

### Phase 4：八字增强（104 个 commit 的大分支）

在八字引擎基础上做了三层扩展：

**神煞引擎（25+ 查找表）：**

- 年支维度：三合（6 种模式通过 `checkSanHeBranch()`）
- 日干维度：禄神、羊刃、天乙贵人、太极贵人、文昌贵人、学堂、词馆、金舆、福星贵人
- 月支维度：天德贵人、月德贵人、血刃、勾绞
- 日支维度：天赦、十恶大败、魁罡
- 通用维度：空亡、红鸾、天喜、丧门、吊客、孤辰、寡宿、元辰

**流年引擎（11 年跨度）：**

- 评分算法：基准 50 + 喜用神/忌神 + 地支关系（六合/六冲/三刑/六害/六破）+ 神煞
- 总结文本：规则模板拼接
- 月干：五虎遁（年上起月法），月份边界通过节气精确判定
- 流年神煞：三维度覆盖（年支→流年、日干→流年、月支→流年）

**卜筮持久化 API：**

- POST /api/divinations（保存，每 profile 每分钟限流 10 次）
- GET /api/divinations?type=bazi（列表，排除 result_data 节省带宽）
- GET /api/divinations/[id]（详情，校验归属）

### 安全加固

- PIN 升级：4 位数字 → 6-20 位字母数字
- httpOnly cookie：彻底去掉 localStorage 认证，删了 301 行样板代码
- 单会话强制：同一 profile 只能有一个活跃 session
- CSP nonce：server/plugins/csp.ts 运行时注入随机 nonce
- HSTS：2 年 max-age + preload
- 速率限制：内存限流，按 profile + endpoint 键控
- 安全日志：登录失败、登出、session 创建都记录

### 性能优化

- 节气计算从 O(n) 年循环改成 O(1) 公式
- 大运查找从 O(n) 线性扫描改成 O(1) 直接索引
- 深拷贝从 JSON round-trip 改成 structuredClone
- 神煞查找表提升到模块作用域，避免每次调用重新分配
- SQLite 写入 5 秒防抖节流
- 登录失败计数查询加复合索引

## 做得不好的地方

### 1. 重复劳动严重

477 个 commit 里有一半是在修"我以为做完了其实没做完"的东西。

**Auth 系统来回折腾（3 次）：**

1. 纯 localStorage
2. 加了 httpOnly cookie 做 fallback，localStorage 优先
3. 改成纯 cookie 模式，删了 301 行样板代码

**Auto-fill banner 改了 3 版：**

1. 简单 emoji 横幅
2. redesign 加 prominent button（中间还忘删 debug console.log、scope 搞错、clock hour 没转时辰 index）
3. 墨韵风格三态设计

**BaziGrid 布局反复（至少 5 次）：**
card → table → 加右侧栏 → 删右侧栏 → 又加回来 → 又删了改朱砂签

**首页布局至少 3 次推翻：**
硬编码数据 → artifact-display card → 50/50 布局

### 2. 格式化 commit 浪费

"fix prettier formatting" 出现 4 次，"fix ESLint errors" 出现 3 次，"resolve 245 ESLint warnings" 一次就冒出 245 个。这些本该在 pre-commit hook 里自动做，直到 6 月 5 日才加上 lint-staged。

### 3. a11y 反复修

WCAG 对比度修了至少 5 轮（ink-light、gold、wuxing-metal 反复调暗），ARIA 标签补了至少 8 次，键盘导航修了至少 4 次。40 个 a11y 相关 commit——说明一开始就没考虑 accessibility。

### 4. 规则引擎的"隐形 bug"

这些 bug 的共同特点是：代码看起来对，但算出来不对。不是语法错误，是逻辑错误——需要对照古籍才能发现。

| Bug                 | 影响                 | 根因                                         |
| ------------------- | -------------------- | -------------------------------------------- |
| 福星查表 = 禄神查表 | 福星在错误位置触发   | 复制粘贴忘了改数据                           |
| 三才评分 +3 应为 +4 | 所有三才评分错误     | 被生 vs 被克的偏移搞混                       |
| 六神偏移 4 位       | 青龙变白虎           | 基准日搞错（2000-01-01 是戊不是甲）          |
| 大运十神始终"—"     | 大运天干无十神       | 收到地支字符而非藏干                         |
| 流年中性文案错      | 中性年显示"谨慎"     | 模板分支少了 NEUTRAL 类型                    |
| 六冲和六合冲突      | 巳申同时 +10 和 -15  | 缺少冲突压制逻辑                             |
| 小寒年份错误        | 1 月出生年柱可能错   | 用错年份查节气                               |
| 姓名 isGenerate +3  | 所有三才评分偏差     | 被生应该是 +4 不是 +3                        |
| 比和缺失            | 同元素和谐默认为凶   | 逻辑分支遗漏                                 |
| 季节调候重复        | 金同时出现在喜忌列表 | prepend 到 favorable 但没从 unfavorable 移除 |

### 5. 功能贪多

做了 11 个工具，每个都"能用但粗糙"。用户反馈"算的不准"——不是引擎算不对，是缺少上下文（命签给单身用户看牛郎织女）。后来决定线上隐藏紫微/合婚/梅花，保留 8 个核心工具。

### 6. 缺少用户上下文

传统算命是面对面先问事再算。线上缺少这个环节，导致签文跟用户状态脱节。今日命签按日期 hash 随机抽取，不考虑用户是否单身、关心什么。今日穿衣的五行建议跟用户个人喜忌无关。

### 7. 移动端适配不足

代码审计能发现 pattern 问题（padding、overflow、truncate），但真机体验只有用户能看到。29 个移动端问题中，6 个 HIGH（"注"面板溢出、表格 min-w 反转、HistoryModal 三层 p-8 嵌套），到现在还没修。

### 8. 没有流程直到翻车

前三个阶段没有审计流程，写完就交。问题反复冒出来，同一块代码改 3-5 遍。直到 talisman-card 组件翻车（5+ 轮重写），才催生了 v4.2 工作流。

## 学到了什么

**技术层面：**

- 多子 Agent 协作：从混乱派发到精确范围 + 验收标准
- 规则引擎：把古籍文字变成可执行代码，需要逐条对照文献
- 设计系统：从零搭建到 13 个页面统一
- 安全：PIN 哈希、httpOnly cookie、CSP nonce、单会话强制
- 测试：2043 测试用例，穷举覆盖组合边界
- 部署：阿里云 + 宝塔 + PM2 + Nginx + SSL + DNS
- 性能：O(n) → O(1) 算法优化

**流程层面：**

- 没有审计 = 反复返工
- 没有设计先行 = 布局反复推翻
- 没有 pre-commit hook = 格式化 commit 浪费
- 没有验收标准 = agent 修一个问题引入一个新问题
- 翻车不可怕，翻车后没有流程改进才可怕

**认知层面：**

- 做产品比写代码难多了
- 代码可以靠 AI 辅助，但用户体验只有真实用户能告诉你
- 477 个 commit 里有一半是在修"我以为做完了其实没做完"的东西
- 但这些反复就是学习成本

## 工作流的演变

| 阶段     | 时期          | 特点                                    |
| -------- | ------------- | --------------------------------------- |
| 无流程   | Phase 1-3     | 写完就交，没有审计，同一块代码改 3-5 遍 |
| 发现阶段 | Phase 4       | 开始做 code review，但标准不统一        |
| 翻车阶段 | talisman-card | 组件重写 5+ 轮，才意识到是流程问题      |
| v4.2     | 6 月 8 日     | 五阶段工作流 + 四条铁律                 |

v4.2 五阶段：分析→派发→审计→报告→提交。
四条铁律：不懂就问不猜、子 Agent 永不提交、文档同步是验收标准、无报告不提交。

## 感受

14 天，480 个 commit，64137 行代码，一个人全栈。从"练练 AI 开发能力"到"认真做一个产品"。

今晚把项目分享给朋友们看，反响让我也不太舒服。大部分都在说"算的不准"，只有最开始跟我聊玄学的那个朋友说了句"不错"。

但朋友们说的是对的。真正的朋友是要批评的，不是一味阿谀奉承。虽然确实没人夸我哈哈。

回头看，最珍贵的不是上线的产品，是这 480 个 commit 里踩过的每一个坑。福星查表复制粘贴错误、三才评分公式正负反了、六神偏移 4 位——这些"看起来对但算出来不对"的 bug，教会了我什么叫"真正检查过"。

还有那些反复推翻的布局、反复重写的 banner、反复修改的 auth 系统——教会了我什么叫"先想清楚再动手"。

做东西就是这样。不是写完代码就完了，是从引擎到 UI 到部署到用户反馈的完整闭环。480 个 commit 就是 480 次"发现问题→解决问题"的循环。

## 个人反思

### 对玄学的看法变化

做这个项目之前是兴趣——看看每天的运势，觉得好玩。做这个项目的期间是享受——每一个新功能的实现、每一个好看的 UI 界面设计出来，都有成就感。做完之后发现，其实受众很小，而且自己也没做好。

### 对 AI 辅助开发的理解

AI 辅助开发真不是口头上说说就能完成的。需要自己的总结和经验，需要知道怎么驾驭 AI。AI 没有视觉、语音描述不清楚——这些都是很大的障碍。不是"让 AI 写代码"就完了，是怎么让 AI 写出你想要的代码。

### 如果重新来过

可能还会做。只不过一定会调研清楚，然后使用更好的工作流。不是边做边摸索流程，而是先把流程想清楚再动手。

### 最骄傲的部分

整体的 UI 设计风格。虽然一直在修修改改，但最后风格挺统一的——墨韵风格从第一个组件到最后一个页面，是一致的。

还有紫微斗数。那个天宫图真是让我用了无数的办法，虽然到最后我也不知道天宫图的展示合不合理……但那个过程很深刻。
