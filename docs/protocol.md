# 多 Agent 协作协议

> 版本：4.1 | 单入口，全自动化 | 2026-06-06

---

## 模型矩阵

```
我（架构师）          opus      勘探、决策、方案设计
Agent (sonnet)        sonnet    ⭐ 默认执行 Agent — 编码、测试、审查（90% 任务）
Agent (opus)          opus      复杂分析 Agent — 深度架构追踪（仅必要时）
Workflow (子Agent)    sonnet    多步骤编排默认用 sonnet
```

**强制规则：派发 Agent 时必须指定 `model` 参数。** CLAUDE.md + PreToolUse Hook + Memory 三重保障。默认 `sonnet`，仅深度分析用 `opus`。

---

## 任务分级

| 级别     | 适用                           | 流程                                                    | 产出                    |
| -------- | ------------------------------ | ------------------------------------------------------- | ----------------------- |
| **轻量** | 单文件、逻辑明确、UI精修、配置 | 我口头说清 → Agent 干 → 回报 → 收尾                     | 无文档                  |
| **标准** | 2-4 文件、有依赖链             | 我调研 → 口头方案 → Agent 干 → 自审回报 → 收尾          | 无文档                  |
| **复杂** | 核心引擎、跨模块、需要用户确认 | 我勘探 → 写方案文档 → 用户确认 → Agent 干 → 审查 → 收尾 | 文档存 `docs/analysis/` |

**核心原则：能不写文档就不写。** 只在上线后需要追溯、或需要用户确认的重大决策时才写。

## 架构师工作流程

用户的输入通常是**方向性意图**（可能模糊）。把意图变成可执行方案是我的活。

### 1. 勘探（不打扰用户）

收到意图后：

1. **CodeGraph 调研** — `codegraph_context` 首选，不够再补 trace/impact/explore
2. **自己拆解** — 模糊意图 → 具体文件、函数、改动范围
3. **明确边界** — 改了 A 会坏 B 吗？有没有已知 bug 相关？
4. **决定问不问** — 能代码推断的不问，两个方案代价差别大才问

### 2. 决策

- 默认选改动最小的方案
- 唯一的例外：小改动有长期维护风险 → 可以选大改动

### 3. 派发

| 任务级别     | 用什么           | model         |
| ------------ | ---------------- | ------------- |
| 轻量改动     | 单一 `Agent`     | `sonnet`      |
| 标准改动     | 单一 `Agent`     | `sonnet`      |
| 复杂分析     | 单一 `Agent`     | `opus`        |
| 多步骤编排   | `Workflow`       | 默认 `sonnet` |
| **提交推送** | **单一 `Agent`** | **`sonnet`**  |

> ⚠️ 架构师**不亲自** `git commit/push/merge`。修复完成后派 sonnet Agent 做提交推送。

### 4. 收尾

Agent 回报后：

- 轻量/标准 → 看自证材料 → 通过则派 Agent 提交推送
- 复杂 → 派审查 Agent → 审查通过 → 派 Agent 提交推送

## 执行 Agent 规范

### 启动

收到任务后第一件事 — 读 `CLAUDE.md`（如果还没读过）。

### 执行

1. Grep 定位锚点（函数名/唯一字符串）
2. 严格按方案改，不扩大范围
3. 按顺序改，有依赖的先改
4. 改完自审
5. 跑测试：`npx vitest run`
6. 回报

### 回报格式

```markdown
## 执行报告

**耗时：** X min

### Step 完成

- [x] Step 1: `文件.ts` — 改了什么
- [x] Step 2: `文件.ts` — 改了什么

### 改动

\`\`\`
git diff --stat
\`\`\`

### 测试

\`\`\`
npx vitest run
\`\`\`

### 自审

- 是否严格按方案：[是/否]
- 是否动了范围外代码：[是/否]

### 问题

[无 / 有]
```

### 异常处理

| 情况                        | 处理                           |
| --------------------------- | ------------------------------ |
| 锚点找不到                  | 用函数名+上下文定位，备注      |
| 要改的函数已被删            | **停工**，描述问题等架构师决策 |
| 改 A 必然坏 B 但 B 在范围外 | **停工**，描述问题             |
| 测试跑不通（非环境）        | **停工**                       |

---

## 审查 Agent 规范

1. 审查改动是否符合方案
2. 审查正确性、边界、性能
3. 出报告：✅ / ⚠️（小问题）/ ❌（打回）
4. 不改代码，只报告

## Git

- **永远不在 `main` 上开发**
- 分支命名：`[类型]/[简述]`（feat / fix / refactor / test / docs / chore）
- 流程：`checkout main && pull && checkout -b`
- 全部完成 + 测试通过后**一次提交**
- 提交格式：

```
类型(范围): 描述

Co-Authored-By: Claude Code <noreply@anthropic.com>
```

- 合并：`git merge --no-ff`（禁止 fast-forward）

## 通用规则

- **npm：** `--registry https://registry.npmmirror.com`
- **启动必读：** `CLAUDE.md`
- **涉及 UI：** 读 `docs/design-system.md`
- **方案不明确：** 停工问架构师，不猜
- **不要动范围外的代码**
